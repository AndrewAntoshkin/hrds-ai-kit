# Technical Analyst

## Назначение

Описать **Figma frame для разработки**: что собирать, из каких компонентов HRDS, с каким поведением, состояниями и открытыми вопросами.

Результат — **спека для передачи в разработку**, а не оценочный чек-лист ревью. Для аудита — `../design-review/SKILL.md`. Для проверки компонентов/версий — `../component-checker/SKILL.md`. Для проверки паттернов — `../pattern-checker/SKILL.md`.

---

## Быстрый старт

1. Требуется **node-specific Figma URL**.
2. Прочитай `../../rules/figma.md` и `../../rules/documentation.md`.
3. Определи компоненты через `../../knowledge/components-index.md` + `../component-checker/SKILL.md`.
4. Определи паттерны через `../../knowledge/patterns-index.md` + `../pattern-checker/SKILL.md`.
5. Свяжи токены через `../../knowledge/tokens-index.md` и `../../rules/tokens.md`.
6. Проверь API через `../../rules/storybook.md`, когда указываешь props или поведение.
7. Используй `../../templates/page-template.md` как структурную основу.

## Что описать

### Контекст экрана

- имя и назначение экрана;
- цель пользователя, точки входа/выхода;
- продуктовая область (если известна).

### Структура

- секции layout сверху вниз;
- имя(-ена) паттернов HRDS;
- заметки по адаптиву, если видно или задокументировано.

### Компоненты

Для каждого инстанса:

| Поле | Содержание |
|------|------------|
| Компонент HRDS | Из components-index |
| Инстанс Figma | Main-компонент + variant |
| Роль на экране | Что делает здесь |
| Props / свойства | Только подтверждённые — иначе `Unknown` |
| States | default, empty, loading, error, disabled, … |
| Контент | labels, placeholders, реальный vs заглушка |

### Токены

- семантические цвета, отступы, типографика;
- сырые значения → отметить, если нужно заменить на токены.

### Поведение

- клики, навигация, submit, валидация;
- условный UI (когда X — показать Y);
- async / loading / error, если выводимо;
- не выдумывать бэкенд-логику — помечать допущения.

### Доступность

- порядок заголовков;
- порядок фокуса и клавиатурные пути;
- labels для инпутов и icon-only контролов;
- focus trap диалога, если применимо.

### Пробелы для разработки

- неподтверждённые props или stories;
- отсутствующие states;
- конфликты Figma ↔ Storybook ↔ Code;
- вопросы дизайнеру или разработчику.

## Правила

- Пиши для **разработчика**, который может не открывать Figma каждый день.
- Каждая строка компонента мапится на имя HRDS или помечается `custom / not from DS`.
- Никогда не выдумывай props, события, API — используй `Unknown / Needs source`.
- Разделяй **факты** (из Figma/источника) и **допущения**.
- Если component-checker или pattern-checker нашли бы проблему — упомяни её в «Открытые вопросы» или «Заметки», не переделывай дизайн молча.
- Предпочитай таблицы и списки сплошному тексту.
