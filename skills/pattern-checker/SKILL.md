# Pattern Checker

## Назначение

Найти, где на экране использован **не тот паттерн HRDS**, и подсказать **правильный** паттерн из дизайн-системы.

Паттерн — это переиспользуемая композиция под пользовательскую задачу (Page Header, Data Table, Empty State, Form Flow и т.д.). Это не отдельный компонент.

Этот скилл **не** проверяет версии компонентов — для этого `../component-checker/SKILL.md`.

## Ход работы

### 1. Определи пользовательскую задачу

Что пользователь делает на этом экране или в секции?

- смотрит список;
- создаёт или редактирует сущность;
- подтверждает необратимое действие;
- попал на пустой результат;
- навигируется по контексту страницы.

### 2. Определи используемый паттерн

Какой паттерн реализует текущий дизайн (намеренно или случайно)?

Назови его из `patterns-index.md` или пометь `custom / no pattern`.

### 3. Определи нужный паттерн

Какой паттерн HRDS **подходит задаче**?

По `patterns-index.md`: when to use / when not to use, обязательные и опциональные компоненты, related.

### 4. Сравни и реши

| Решение | Когда |
|---------|-------|
| `correct pattern` | Верный паттерн под задачу |
| `wrong pattern` | Другой индексный паттерн подходит лучше |
| `pattern vs component` | Задача решается одним компонентом, не паттерном |
| `custom screen` | Продуктовый layout, не общий паттерн HRDS |
| `needs review` | Паттерн зависит от неподтверждённых компонентов или задача неясна |

## Частые несоответствия

Явно отмечай, когда применимо:

| Используется (неверно) | Нужно | Почему |
|------------------------|-------|--------|
| Content Header | Page Header | Контекст уровня страницы, breadcrumbs, primary action |
| Page Header | Content Header | Секция внутри страницы, не полный chrome страницы |
| Empty State | Error state | Нет данных vs упавшая операция |
| Только Dialog | Confirmation Flow | Необратимое действие требует полного паттерна подтверждения |
| Кастомная таблица | Data Table | Табличные данные с sort/filter/actions |
| Отдельные поля формы | Form Flow | Создание/редактирование с валидацией и submit |
| Сетка карточек | List / Data Table | Зависит от задачи: обзор vs сравнение колонок |

Добавляй продуктовые несоответствия только когда подтверждены — не выдумывай новые имена паттернов.

## Правила

- Всегда называй **оба**: используемый паттерн и рекомендуемый.
- Объясняй несоответствие через **пользовательскую задачу**, не через вкус.
- Если верный ответ — **один компонент**, ставь `pattern vs component` и укажи на component-checker.
- Не предлагай новый общий паттерн HRDS без `needs review`.
- Если паттерн продуктовый — пометь `custom screen`, не навязывай запись в индекс.
- Перечисли обязательные компоненты рекомендуемого паттерна из индекса.
