Каждый экран без контекста снова забирает время.

Задача звучит просто: «опиши экран по HRDS». На практике агент выдумывает variants, путает паттерны, не видит отвязанные инстансы и не сверяет props со Storybook. Разработчик получает спецификацию с догадками. Дизайнер снова видит те же замечания на ревью.

Kit нужен, чтобы не объяснять агенту одни и те же правила заново. Он даёт маршрут проверки и честно помечает места, где источника пока нет.

Что это и зачем

HRDS AI Kit — не Figma, не Storybook и не npm-пакет. Это контекст для AI-агента: ссылки, индексы, правила и skills, которые ведут к проверяемым источникам. Storybook отвечает «какой API у Button». Kit отвечает «где это проверить и как пометить неизвестное».

01

Что внутри

knowledge/ — источники, индексы и журнал решений. rules/ — ограничения для агента. skills/ — готовые маршруты. examples/ — примеры формата.

02

Зачем командам

Без kit агент выдумывает props, путает паттерны и пропускает устаревшие инстансы. Kit снижает число догадок, но не отменяет человеческое ревью.

03

Кому

Дизайнеры проверяют макеты и версии инстансов. Разработчики получают спецификацию. Команда дизайн-системы пополняет индексы.

Можно не начинать с нуля.

HRDS AI Kit собирает рядом с проектом то, что обычно приходится объяснять агенту вручную: какие источники важнее, где искать компоненты, какие данные уже проверены, а какие требуют ревью. Это пилот v0.2, не каталог на замену Storybook.

  1. 001 / Сначала источники

    Главные ссылки лежат в knowledge/sources.md. Для описания макета нужен Figma URL с конкретным node-id. Props и поведение проверяются в Storybook 13.3.0 или коде.

  2. 002 / Связка компонента

    Цепочка фиксируется явно: свойство в Figma → control в Storybook → import в коде → пример использования. Если источника нет, агент пишет Needs review.

  3. 003 / Для работы с агентом

    Агент не должен заново угадывать правила HRDS. В kit есть rules/, skills/ и обёртки .cursor/skills/ для вызова через / в Cursor.

  4. 004 / Проверка версий

    component-checker помогает найти отвязанные, устаревшие и чужие инстансы. Проверка идёт относительно библиотеки 13.3.0 и журнала решений.

  5. 005 / Проверка паттерна

    pattern-checker помогает понять, подходит ли выбранный паттерн: Page Header или Content Header, Empty State или Error, список или форма.

  6. 006 / Спецификация для разработки

    technical-analyst готовит спецификацию: компоненты, props, состояния, токены, поведение и открытые вопросы. Непроверенное помечается отдельно.

  7. 007 / Пробелы вместо догадок

    Starter и Proposed явно помечены. Confirmed ставится только для проверенных фактов. Props не выдаются за правду без Storybook или кода.

  8. 008 / Подключение за три шага

    Скачать → добавить папку в workspace → вызвать /component-checker или /design-review с Figma URL, где есть node-id.

002 / Метки в выводе

Confirmed

Есть источник: метаданные в Figma, story в Storybook или код. Можно опираться, но перед запуском в продукт всё равно сверить.

Starter

Данные есть в kit, но ещё не проверены по источникам. Агент не должен выдавать их за факт.

Needs review

Prop или variant без подтверждённого источника. Нужно найти story или код, а не додумывать.

Proposed

Гипотеза по примеру формата. Это не факт. Рядом всегда должно быть «сверить в Storybook».

003 / Приоритет источников

Если источники расходятся, порядок такой. Агент фиксирует конфликт, а не решает его молча.

  1. 01Код / Storybook — props, controls, поведение
  2. 02Компонент в Figma — структура, variants, properties
  3. 03Rules — доступность, naming, ревью
  4. 04Examples — формат вывода, не источник правды
  5. 05Decision log — принятые исключения

Перед подключением

Q.001 / Это замена Figma или Storybook?

Нет. Kit — контекст для AI-агента. Он указывает на источники и правила. Дизайн остаётся в Figma, API — в Storybook и коде.

Q.002 / Чем отличается от «просто скинуть ссылку на Storybook»?

Storybook показывает API, но не объясняет агенту, что и в каком порядке проверять. Kit фиксирует приоритет источников, статусы Starter/Confirmed и готовые маршруты для ревью и спецификации.

Q.003 / Работает ли с Cursor?

Да. Skills подключены через .cursor/skills/. Вызов: /component-checker, /pattern-checker, /technical-analyst, /design-review, /prototype-builder.

Q.004 / Можно ли генерировать код для продакшена?

Только после человеческого ревью. Для кода нужно подтвердить props в Storybook или исходниках. Kit снижает число догадок, но не снимает ответственность с команды.

Q.005 / Что значит Starter?

Это данные, которые уже есть в kit, но ещё не подтверждены по Figma, Storybook или коду. Агент должен помечать их как непроверенные.

Q.006 / Какая версия библиотеки?

Актуальный Storybook: 13.3.0. Пакет: @yandex-int/hr-components. Проверку устаревших инстансов запускай через /component-checker.

Q.007 / Что kit не делает?

Это не полный каталог HRDS, не автопилот и не замена ревью дизайн-системы. Если prop не подтверждён в Storybook или коде, kit не делает его настоящим.

Q.008 / Куда писать новые данные?

Ссылку на источник — в sources.md. Имя компонента или паттерна — в индекс. Решение — в decision-log.md. Правило для агента — в rules/. Развёрнутое описание — в examples/ по шаблону.

Q.009 / С чего начать на реальном экране?

Открой макет в Figma → скопируй URL с node-id → запусти /design-review или /component-checker → исправь замечания → запусти /technical-analyst для спецификации. Сценарии лежат в examples/scenarios.md.

Q.010 / Кто поддерживает kit?

Команда дизайн-системы и платформы поддерживает индексы и переводит записи из Starter в Confirmed. Продуктовые команды используют skills и закрывают Needs review по Storybook или коду. v0.2 — пилот: 5 Confirmed components и 5 Starter patterns.
